## Fix: nested collection.where().exists() in WHERE clauses

### Bug

Using `collection.where(item => item.refNav.nestedCollection.where(...).exists()).exists()` in a WHERE clause crashed with `TypeError: condition.getFieldRefs is not a function`.

This pattern is common for filtering by existence through navigation chains, e.g.:

```typescript
db.products.where(p =>
  p.b2bGroupProducts
    .where(gp => gp.b2bGroup.b2bPartnerGroups
      .where(pg => eq(pg.b2bPartnerId, partnerId))
      .exists())
    .exists()
)
```

### Cause

`CollectionQueryBuilder.exists()` returns `this` cast as `SqlFragment<boolean>`, but `CollectionQueryBuilder` did not implement the `Condition` interface methods (`getFieldRefs()`, `buildSql()`) that the query builder requires when processing WHERE conditions.

### Fix

Added `getFieldRefs()` and `buildSql()` to `CollectionQueryBuilder` so it can act as a proper condition via duck typing. The `buildSql()` method produces a correlated `EXISTS (SELECT 1 FROM ... WHERE ...)` subquery that handles:

- Navigation path JOINs (reference navigations like `gp.b2bGroup → b2b_groups`)
- SelectMany JOINs (intermediate table traversal)
- Correlation to parent table (`fk = parent.id`)
- Additional WHERE conditions with correct alias rewriting
- Recursive nesting (inner `.exists()` conditions)

### Files Changed

- `src/query/query-builder.ts` — added `getFieldRefs()` and `buildSql()` to `CollectionQueryBuilder`
- `tests/queries/nested-collection-where-exists.test.ts` — new test covering the pattern in WHERE and SELECT contexts
