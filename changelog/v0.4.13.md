## Fix: multi-level navigation in WHERE clauses losing intermediate JOINs

### Bug

Using multi-level navigation properties in WHERE conditions (e.g., `eq(o.task!.level!.name, "foo")`) could fail to resolve intermediate JOINs, producing invalid SQL that referenced aliases without corresponding JOIN clauses.

### Cause

When `createMockRow()` builds FieldRefs in WHERE context (`preserveOriginal: true`), it only preserved `__dbColumnName`, `__tableAlias`, and `__mapper`. The `__navigationAliases` and `__sourceTable` properties were dropped, so the two-phase JOIN detection (`collectTableAliasesFromSelection`) could not see intermediate navigation steps and skipped generating the required JOINs.

### Fix

- Preserve `__navigationAliases` and `__sourceTable` on FieldRefs created in WHERE context, so intermediate tables in multi-level navigation chains are correctly collected and joined
- Ensure `collectTableAliasesFromSelection` also collects intermediate navigation aliases from condition FieldRefs, not just top-level selection fields

### Files Changed

- `src/query/query-builder.ts` — preserve navigation metadata on WHERE FieldRefs; collect intermediate aliases from conditions
