## Fix: nested scalar-subquery summand in `.sum()` / `.min()` / `.max()`

### What

`.sum()` (and `.min()`, `.max()`) now accept a nested scalar subquery as their argument inside projections, regardless of the active `collectionStrategy` (`lateral`, `cte`, or `temptable`). Previously this threw `SUM requires an aggregate field` before SQL generation, even though the shape is valid PostgreSQL.

```typescript
await db.users
  .select(u => ({
    username: u.username,
    totalComments: u.posts!.sum(p =>
      p.postComments!.where(pc => gt(pc.id, 0)).count()
    ),
  }))
  .toList();
```

Emits the expected correlated form:

```sql
SELECT "users"."username",
  (SELECT COALESCE(SUM(
     (SELECT COALESCE(COUNT(*), 0)
        FROM "post_comments" "..."
        WHERE "..."."post_id" = "lateral_0_posts"."id"
          AND "..."."id" > $1)
  ), null)
  FROM "posts" "lateral_0_posts"
  WHERE "lateral_0_posts"."user_id" = "users"."id") AS "totalComments"
FROM "users";
```

### Why it rejected before

`CollectionQueryBuilder.buildCTE` resolved the aggregate argument by invoking the user's selector and extracting `__dbColumnName` from the result. When the selector returned a nested `CollectionQueryBuilder` (a scalar subquery), no column name existed, `aggregateField` stayed `undefined`, and every strategy's scalar-aggregation builder threw on `MIN/MAX/SUM`.

### How the fix routes it

- `CollectionQueryBuilder.buildCTE` detects a nested collection returned by the `.sum/.min/.max` selector, invokes `buildCTE` on it with a forced `collectionStrategy: 'lateral'` override (so the nested build always emits a correlated *scalar* subquery rather than a CTE/temp-table result that can't compose inside `SUM(...)`), and forwards the subquery SQL through a new `aggregateExpression` field on `CollectionAggregationConfig`.
- All four scalar-aggregation builders (correlated lateral, lateral-join, CTE, temp table) now wrap `aggregateExpression` with the aggregate function (`SUM(<subquery>)`) when set, and fall back to the column form (`SUM("col")`) otherwise.
- The field-selection step skips per-column processing when the selector returns a `CollectionQueryBuilder` or `SqlFragment` â€” previously it iterated the builder's own properties as fake columns and polluted the parameter list with garbage values.

### Performance

`CollectionQueryBuilder.buildCTE` previously invoked the user's selector up to three times (field collection, aggregate-argument detection, navigation-join detection), rebuilding mock proxies and any nested builders each time. The selector is now invoked exactly once per call and the result is reused across steps. Navigation-join detection is skipped when the result is a scalar subquery (the walk produces nothing useful for that shape).
