## Feature: `CREATE INDEX CONCURRENTLY` support

### What

Indexes can now be created with PostgreSQL's `CONCURRENTLY` option, which avoids holding a long `ACCESS EXCLUSIVE` lock on the table while the index is built. This is the recommended way to add indexes to large production tables without blocking writes.

Two opt-in modes are provided — per-index and global — both off by default so existing behavior is unchanged.

### Per-index opt-in: `.concurrent()`

Mark individual indexes to be built concurrently:

```typescript
entity.hasIndex('idx_items_email', e => [e.email]).concurrent();

entity.hasIndex('uq_items_name', e => [e.name])
  .isUnique()
  .concurrent();
```

Generates:

```sql
CREATE INDEX CONCURRENTLY IF NOT EXISTS "idx_items_email" ON "items" ("email");
CREATE UNIQUE INDEX CONCURRENTLY IF NOT EXISTS "uq_items_name" ON "items" ("name");
```

### Global opt-in: `getSchemaManager({ concurrentIndexes: true })`

Flip every index over to `CONCURRENTLY` for a single migration run without changing entity definitions:

```typescript
// Force CONCURRENTLY for every CREATE INDEX in this run
await db.getSchemaManager({ concurrentIndexes: true }).ensureCreated();

// Works for incremental migrations too
await db.getSchemaManager({ concurrentIndexes: true }).migrate();
```

Precedence: an index is created concurrently when either the per-index `.concurrent()` flag OR the global `concurrentIndexes` option is set.

### Caveat

PostgreSQL forbids `CREATE INDEX CONCURRENTLY` inside an explicit transaction block. Use it through `DbSchemaManager.migrate()` / `ensureCreated()` (autocommit). The transactional journal-based `MigrationRunner` wraps each migration file in a transaction, so `CONCURRENTLY` statements emitted in scaffolded migrations must be run in autocommit mode by the operator.

### Files Changed

- `src/entity/entity-base.ts` — added `concurrent?: boolean` to `IndexMetadata`
- `src/schema/table-builder.ts` — added `concurrent?: boolean` to `IndexDefinition`
- `src/entity/entity-builder.ts` — added `IndexBuilder.concurrent()` fluent method
- `src/migration/db-schema-manager.ts` — `concurrent` threaded through `create_index` operation, `analyze()`, `migrate()` phase-3 builder, `executeOperation`, `executeCreateIndex` (emits `CONCURRENTLY`), and `describeOperation`; new `concurrentIndexes` option on `DbSchemaManager` constructor that OR-s with the per-index flag
- `src/entity/db-context.ts` — `getSchemaManager({ concurrentIndexes })` on both the base method and the subclass override
- `src/migration/migration-scaffold.ts` — scaffolded SQL emits `CONCURRENTLY` when the flag is set
- `tests/schema/concurrent-index.test.ts` — new test suite (5 tests) covering per-index flag, unique + concurrent, mixed concurrent/plain indexes on the same table, global override, and idempotent re-runs
